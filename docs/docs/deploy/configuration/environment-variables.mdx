# Environment Variables

All configuration is done via environment variables. The authoritative template is [`platform/core/.env.example`](https://github.com/rush-db/rushdb/blob/main/platform/core/.env.example) in the repository.

---

## Required (Self-Hosted)

These four variables must be set or RushDB will not start correctly.

| Variable                        | Default | Description                                                                                                                                                                             |
| ------------------------------- | ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `RUSHDB_SELF_HOSTED`            | —       | Must be `true` to enable self-hosted mode                                                                                                                                               |
| `RUSHDB_LOGIN`                  | —       | Admin account username                                                                                                                                                                  |
| `RUSHDB_PASSWORD`               | —       | Admin account password                                                                                                                                                                  |
| `RUSHDB_AES_256_ENCRYPTION_KEY` | —       | **Exactly 32 characters.** Used for AES-256 API token encryption and JWT signing. Do not change after initial setup without re-encrypting stored data. Generate: `openssl rand -hex 32` |

## AI — Embeddings

Enables vector similarity search. All four variables must be set together; omit them entirely to disable embedding indexes.

| Variable                          | Default                     | Description                                                                                                     |
| --------------------------------- | --------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `RUSHDB_EMBEDDING_BASE_URL`       | `https://api.openai.com/v1` | Base URL of an OpenAI-compatible embeddings endpoint                                                            |
| `RUSHDB_EMBEDDING_API_KEY`        | —                           | Bearer token for the embedding provider                                                                         |
| `RUSHDB_EMBEDDING_MODEL`          | —                           | Model identifier, e.g. `text-embedding-3-small` (OpenAI direct) or `openai/text-embedding-3-small` (OpenRouter) |
| `RUSHDB_EMBEDDING_DIMENSIONS`     | —                           | Number of dimensions the model outputs. Must match the model exactly.                                           |
| `RUSHDB_EMBEDDING_BATCH_SIZE`     | `500`                       | Records per batch during index backfill                                                                         |
| `RUSHDB_EMBEDDING_MAX_RUNTIME_MS` | `50000`                     | Max milliseconds the backfill scheduler runs per tick                                                           |

:::caution Latency matters
Semantic search embeds the user's query on the request path — pick a model/provider with consistent sub-second latency. Models served without capacity guarantees can take tens of seconds per embedding, which makes query-time search unusable regardless of index state.
:::

:::caution Changing the model requires re-embedding
Vectors produced by different models are incompatible even at identical dimensions, so after `RUSHDB_EMBEDDING_MODEL` changes every managed index must be re-embedded.

This happens **automatically**: on startup the server detects managed indexes keyed to a different model, strips their old vectors, and resets them to `pending` for the backfill scheduler to re-embed — so a normal deploy that changes the model needs no manual step. Search on affected indexes returns partial/empty results until re-embedding completes. Rolling deploys are safe: instances only backfill indexes keyed to their own configured model, so pods still running the old model cannot write stale vectors into re-keyed indexes.

To preview the impact before deploying (or re-drive a failed startup run), use the script from `platform/core`:

```bash
pnpm embeddings:migrate-model --dry-run   # preview affected indexes
pnpm embeddings:migrate-model             # strip old vectors, queue re-embedding
```

Changing `RUSHDB_EMBEDDING_DIMENSIONS` is not covered — indexes with a different dimension count must be deleted and recreated.
:::

## Managed Connectors / synx

These variables are required only when running managed continuous-sync connectors. The RushDB platform process stores connector configuration, encrypted secrets, offsets, and status; synx workers run the CDC pipelines and call the platform's private connector-control endpoints.

### Platform/core

| Variable                          | Default | Description                                                                                                   |
| --------------------------------- | ------- | ------------------------------------------------------------------------------------------------------------- |
| `RUSHDB_SYNX_CONTROL_TOKEN`       | —       | Shared internal token accepted by `/api/v1/connectors/_internal/*`. Required when synx workers are enabled.   |
| `RUSHDB_SYNX_DESTINATION_API_KEY` | —       | Internal write API key passed to synx assignments. Bootstrap option until per-project connector tokens exist. |

### synx worker

| Variable                    | Default | Description                                                              |
| --------------------------- | ------- | ------------------------------------------------------------------------ |
| `RUSHDB_CORE_INTERNAL_URL`  | —       | Base URL of the RushDB platform/core process, e.g. `http://rushdb:3000`. |
| `RUSHDB_SYNX_CONTROL_TOKEN` | —       | Same token configured on platform/core.                                  |
| `RUST_LOG`                  | `info`  | synx log verbosity.                                                      |

Run synx in managed mode:

```bash
synx managed \
  --control-url "$RUSHDB_CORE_INTERNAL_URL" \
  --token-env RUSHDB_SYNX_CONTROL_TOKEN \
  --worker-id "$HOSTNAME" \
  --claim-interval-ms 5000 \
  --heartbeat-interval-ms 20000 \
  --lease-ttl-ms 60000
```

Managed workers run as daemons: they claim one runnable connector, renew its lease while the CDC pipeline is active, release the lease on shutdown, and continue polling for the next assignment. Platform/core reclaims expired leases so another worker can pick up a connector after a crash.

## Advanced

### MCP OAuth — RS256 JWT Signing

By default MCP OAuth tokens are signed with the symmetric `RUSHDB_AES_256_ENCRYPTION_KEY`. For production deployments that need JWKS discovery (RS256), set an asymmetric key pair instead.

| Variable                        | Default            | Description                                                                          |
| ------------------------------- | ------------------ | ------------------------------------------------------------------------------------ |
| `RUSHDB_JWT_PRIVATE_KEY`        | —                  | PEM-encoded RSA private key (multiline). Prefer the `_BASE64` variant in containers. |
| `RUSHDB_JWT_PUBLIC_KEY`         | —                  | PEM-encoded RSA public key                                                           |
| `RUSHDB_JWT_PRIVATE_KEY_BASE64` | —                  | Base64-encoded PEM private key — easier to pass in env without escaping              |
| `RUSHDB_JWT_PUBLIC_KEY_BASE64`  | —                  | Base64-encoded PEM public key                                                        |
| `RUSHDB_JWT_KID`                | `rushdb-mcp-rs256` | Key ID published in `/.well-known/jwks.json`                                         |

Generate a key pair:

```bash
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem
# Base64-encode for containers:
base64 -i private.pem | tr -d '\n'   # → RUSHDB_JWT_PRIVATE_KEY_BASE64
base64 -i public.pem  | tr -d '\n'   # → RUSHDB_JWT_PUBLIC_KEY_BASE64
```
